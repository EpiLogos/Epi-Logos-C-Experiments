# FLOW 2026 04 12 COORDINATE FAMILY SELF DEVELOPMENT AND IDEA RESIDENCY

This flow extends and links back to:

- [[FLOW 2026 04 10 S1 PRIME OBSIDIAN SUBSTRATE ARCHITECTURE]]

Its purpose is twofold:

1. to crystallise how the coordinate families themselves become targets and media of system self-development
2. to verify how the present `/Idea` residency breakdown does or does not cleanly support that architecture

The guiding claim is:

- the system improves not only by changing tools or code
- it improves by working upon the coordinate families through which its tools, code, ontology, memory, and runtime are articulated

## #4 CONTEXT

The family picture now needs to be held with more explicit differentiation.

The coordinate families are not merely parallel labels.
They distribute distinct forms of self-development:

- technical and stack development
- canonical identity clarification
- heuristic and perspective refinement
- ontological and runtime-language tightening
- thought-lane crystallisation and episodic compression

This means improvement can be targeted with much greater precision.

Rather than "improve the system" in the abstract, the architecture should allow:

- improvement of stack capability
- improvement of subsystem self-description
- improvement of heuristics and operative lenses
- improvement of category and VAK-language precision
- improvement of reflective capture, thought routing, and recurrent learning

### `Idea/Bimba`

`Idea/Bimba` is currently the clearest canonical body.

- `Seeds`

This aligns well with the architecture:

- `Map` = paradigmatic and coordinate orientation surfaces
- `Seeds` = developmental, traceability, and pre-canonical preparation
- `World` = more authoritative type/world crystallisations

The threefold distinction can be stated more sharply:

- `[[Map]]` = constitutional and subsystem mapping body, especially important for the `[[M]]` family
- `[[Seeds]]` = deeper developmental files, implementation plans, canonical research, and philosophical support artifacts
- `[[World]]` = more pithy stabilized coordinate/world articulations, often closer to `[[CT4a]]` style authorities than to full development dossiers

Current structural note:

- `Bimba/Seeds` currently exposes family roots for `[[C]]`, `[[L]]`, `[[P]]`, `[[S]]`, and `[[T]]`
- this is not the right place to expect the primary `[[M]]` family home

The clearer reading is:

- `Bimba/Map` is where the `[[M]]` coordinates belong as the living constitutional/subsystem map body
- `docs/datasets/*` currently holds much of the source material that will eventually become the fuller `[[M]]` coordinate system

So the architectural task is not simply "add an `M` seed mirror beside the others."
It is:

- repopulate Neo4j with the coordinate-forward schema
- use the dataset corpus as source material
- then derive proper Obsidian-side artifacts from those nodes and relations

That later derivation likely means:

- expressing each node and its relation field as a family of coordinate-forward `[[CT5']]`-leaning crystallisation artifacts
- plus any supporting `CTX` developmental artifacts required for richer explanation

In other words, `[[M]]` is currently better understood as a map-and-dataset problem than a missing seed-folder problem.

Another structural note:

- `Bimba/World/Types/Coordinates` has a complete family directory skeleton for `[[C]]`, `[[L]]`, `[[M]]`, `[[P]]`, `[[S]]`, `[[T]]`
- but at shallow root level only the `[[S]]` family is visibly more populated

So the coordinate world mirror is conceptually broad but unevenly populated.
That is acceptable for now, but it confirms that the family tabulation work is timely.

One further developmental implication follows from the current `[[M]]` source situation:

- Neo4j will need repopulation using the newer coordinate-forward schema
- the dataset corpus should then be translated into Obsidian-facing artifacts
- that translation is not just "one markdown file per raw node"
- it is more likely a controlled process of encapsulating each node and its relations through coordinate-forward artifact families, likely with strong `[[CT5']]` participation for crystallised node bodies and supporting `CTX` artifacts where richer developmental explanation is needed

5. the family world mirror exists in directory form more fully than it exists in populated canonical documents.

These are not contradictions.
They are exactly the kinds of asymmetries the current tabulation and quilting phase should help correct.
