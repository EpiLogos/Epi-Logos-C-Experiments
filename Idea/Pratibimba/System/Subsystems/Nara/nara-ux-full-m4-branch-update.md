# Nara UX — Full M4 Branch Update

## 0. Purpose of This Update

This document updates the Nara UX/system framing so that Nara is no longer understood mainly as a journal-plus-oracle interface, nor as a system that continually rewrites a user's identity. The corrected view is:

> **Nara is the protected personal-Pratibimba field of M4.**  
> It receives life as body, journal, dream, oracle, time, symbol, and identity-evidence.  
> It integrates these through QL/MEF.  
> Its real logical-intelligence engine is **M4.3**, which translates pattern across modalities and returns it into the user's body-field, lived field, and personal-Pratibimba.  
> The site of living update is **#4.4.4.4 Personal Pratibimba**, not the source identity-system branch M4-0.  
> M4.5 is the review/canon/teaching relay where Epii may reveal the map when the user's need or interest calls for it.

This update clarifies six points:

1. **M4-0 is a placeholder branch for identity systems**, not the user and not a mutable self-definition engine.
2. **#4.4.4.4 is the living personal-Pratibimba locus** where the protected personal field, quaternionic state, and psychoid rendering are updated.
3. **M4.3 is the true logical-intelligence engine of Nara**, integrating symbolic systems into body, life, dream, memory, and trajectory.
4. **M4.1 and M4.4 are not passive inputs**: the body-field and lived/dream/journal field are the proving grounds of all symbolic patterning.
5. **M4.2 oracle artifacts are dialect events**, not the central intelligence; Tarot, I Ching, astrology, and Mahāmāyā speak through QL/MEF and are integrated by M4.3.
6. **M4.5 is both review gate and Epii relay**: it governs what becomes stable, and it opens the canon/map only when pedagogically appropriate.

# 1. Core UX Axiom

The corrected Nara UX axiom is:

> **Nara does not define the user. Nara maintains the protected mirror-field in which the user's body, life, dreams, symbolic events, and identity-evidence are brought into relation through QL/MEF.**

This prevents three common errors:

- reducing the user to M4-0 identity systems;
- treating Tarot/I Ching/oracle modalities as the intelligence itself;
- allowing daily activity, journal moods, or oracle casts to overwrite stable identity.

The full M4 branch should be read as a living organism:

```text
M4-0  Identity-system evidence       → source layers, not the user
M4-1  Body / transit / somatic field  → embodied condition now
M4-2  Oracle / symbolic dialects      → event-level symbolic utterance
M4-3  Transformation intelligence     → logical integration engine
M4-4  Life / dream / journal field    → lived evidence and memory
M4-5  Epii / Sophia relay             → review, canon, teaching, governance

#4.4.4.4 Personal Pratibimba        → living protected personal field
```

The system does not ask, “What is the user?”  
It asks:

> **How is this lived event altering the relation between identity-evidence, body-state, symbolic utterance, transformation trajectory, lived memory, and reviewed meaning?**

# 2. The Full M4 Branch

## 2.1 M4-0 — Identity-System Evidence, Not the User

M4-0 is the **Identity Matrix** branch. It contains the structured identity systems:

```text
M4-0-0  Birthdate / name / Pythagorean encoding
M4-0-1  Decanic astrology / Parāśakti chart
M4-0-2  Jungian / 16-type / phenomenal type layer
M4-0-3  Gene Keys / Mahāmāyā 64-code bridge
M4-0-4  Human Design / BodyGraph mechanics
M4-0-5  Identity Quintessence synthesis
```

But M4-0 must be treated as **identity-system evidence**, not as the living identity of the user.

M4-0 is a placeholder and computation branch for sources that can contribute to `Q_identity`, but it is not the user. It is not where journal entries, dreams, body events, oracle casts, or daily transformations rewrite the person.

Correct formulation:

> **M4-0 stores and computes identity-system layer outputs. It does not define the user.**

M4-0 outputs must remain:

- traceable;
- evidence-bearing;
- confidence-bearing;
- modality-specific;
- contradiction-preserving;
- revisable only when source identity-system evidence changes.

Examples of legitimate M4-0 updates:

- user supplies birth time correction;
- a new M4-0-2 Jungian protocol is completed;
- Gene Keys data is computed;
- Human Design bodygraph is added;
- M4-0-5 is rerun because new identity-system layers are available.

Examples of illegitimate M4-0 updates:

- a dream changes the user's “identity type”;
- a Tarot reading rewrites the natal baseline;
- a journal mood mutates M4-0-5;
- Nara treats a symbolic pattern as a final user identity claim.

The M4-0 branch supplies **stable symbolic evidence**. It is not the living update surface.

## 2.2 #4.4.4.4 — The Living Personal Pratibimba

The site that actually receives live update is:

> **#4.4.4.4 Personal Pratibimba**

This is the deepest #4 nesting within Nara:

```text
M4          Nara
M4-4        Context & Lenses
M4-4-4      Phenomenological Lens
M4-4-4-4    Personal Pratibimba
```

This is where the user is represented as a protected, living, contextualised personal field. It is the place where stable identity-evidence, current body-state, live activity, temporal context, and symbolic process compose into a lived personal form.

The corrected rule is:

> **M4-0 contributes stable identity-system evidence. #4.4.4.4 holds the living personal-Pratibimba that is updated through experience, resonance, trajectory, and review.**

So when Nara “learns” from a journal entry, dream, oracle cast, or body-state report, it does not mutate M4-0. It updates the protected personal-Pratibimba state through `Q_activity`, trajectory records, Graphiti episodes, and possibly reviewed augment proposals.

## 2.3 M4-1 — Body, Transit, and Somatic-Medicinal Field

M4-1 is the user's **embodied condition now**.

It includes:

- somatic reports;
- energy, fatigue, sleep, appetite, craving, illness, tension;
- body-zone / chakra / planetary correspondences when available;
- current transits, decanic weather, planetary hour, lunar state;
- elemental or medicinal signals;
- `Q_transit` as the somatic/astrological/temporal current.

M4-1 is where symbolic intelligence must become accountable to embodiment.

Nara should always be able to ask:

> Does this pattern show up in the body?  
> Is the user grounded, exhausted, inflamed, dissociated, restless, heavy, open, or overcharged?  
> Does the reading need a body-practice rather than a conceptual explanation?

M4-1 prevents Nara from becoming purely interpretive. A symbolic pattern is not fully integrated until it has passed through the body-field.

## 2.4 M4-2 — Oracle and Symbolic Dialect Events

M4-2 is the oracle service branch.

It includes:

- Quaternal Tarot;
- Quaternal I Ching;
- dream-omen readings;
- Mahāmāyā hexagram/codon/Tarot bridges;
- Vāma Śaktis contemplative recognition offerings;
- symbolic casts, spreads, readings, and ritual questions.

M4-2 provides **symbolic events**. It is not the whole intelligence.

Tarot and I Ching are dialects of QL/MEF:

- Tarot speaks image, narrative, archetypal scene, suit, number, decan, and court-function.
- I Ching speaks line-state, polarity, moving line, hexagram, transformed hexagram, nuclear interior, and Wu Xing process.
- Mahāmāyā supplies shared symbolic code-space across hexagram, codon, binary, Tarot, and later Gene Keys.

M4-2 events may update `Q_activity`, but only as event-level perturbations with decay, provenance, privacy class, and review status. They must not automatically become identity.

M4-3 is the central correction in this update.

It should be treated as Nara's **real logical-intelligence engine**: the layer that integrates patterning across symbolic systems and returns it into the user's body, life, dream, and personal-Pratibimba.

## 2.6 M4-4 — Life, Dream, Journal, Graphiti, and Evidence

M4-4 is the lived field.

All symbolic interpretation must return to M4-4 for evidence. If a pattern cannot be grounded in lived evidence, it remains a contemplative hypothesis, not a claim.

M4-4 is also where the day-as-episode structure lives. Every artifact should be NOW-stamped, privacy-classed, provenance-bearing, and tied to a DayContainer, not treated as an informal note.

## 2.7 M4-5 / M4.5 — Epii, Sophia, Review, and Canon Teaching

M4.5 is the review and relay gate.

It asks two different but related questions:

1. **Review question:**  
   Should this pattern become reviewed insight, playbook, curriculum, or identity-augment proposal?

2. **Pedagogical question:**  
   Should Epii reveal the map, canon, source, or teaching-path because the user's need or interest calls for it?

This gives M4.5 a dual role:

```text
M4.5 as governance gate:
  proposed → reviewed → accepted | rejected → applied

M4.5 as pedagogical relay:
  personal question → map/canon coordinate → teaching path → return to life
```

Epii should not lecture by default. It enters when teaching serves the user's live process.

The M4.5 rule:

> **Nara mirrors the lived pattern. M4.5 recognises the teaching threshold. Epii conducts the user to the map. The canon teaches. Nara returns the teaching to life.**

# 3. M4-3 as Nara's Logical-Intelligence Engine

## 3.1 Why M4-3 Must Be Central

If Nara is only M4-2 oracle plus M4-4 journal, it becomes a symbolic journaling app.

If Nara is M4-0 identity plus M4-1 body, it becomes a profile and wellness dashboard.

If Nara is M4-5 teaching without M4-3 transformation, it becomes a metaphysical tutor disconnected from lived change.

M4-3 is what makes Nara intelligent:

> **M4-3 integrates the logic of transformation across all symbolic, somatic, temporal, and phenomenological systems.**

This means M4-3 is the layer that can say:

- the Tarot image, I Ching moving line, dream motif, and journal pattern are speaking the same transformation;
- the body-state does or does not support the symbolic interpretation;
- the pattern is recurring across days, not merely present once;
- the current event is activity-level only and should not be promoted;
- the user may need an Epii map-teaching because the same pattern has become a learning threshold.

M4-3 is therefore Nara's **logic of integration**.

# 4. The Correct Update Target: #4.4.4.4, Not M4-0

## 4.1 The Mistake to Avoid

The dangerous simplification is:

```text
journal/dream/oracle event → identity update → M4-0 changes
```

That would make Nara deterministic, unstable, and ethically wrong.

The corrected flow is:

```text
journal/dream/oracle/body event
→ M4-3 PatternPacket
→ Q_activity delta / trajectory update
→ #4.4.4.4 Personal Pratibimba update
→ optional M4.5 review
→ possible reviewed adjustment to protected personal model
```

M4-0 remains the source branch of identity systems. It may provide baseline evidence and stable reference structures, but daily lived activity does not mutate M4-0.

## 4.2 What “Personal Pratibimba Update” Means

A #4.4.4.4 update may include:

- updated `Q_activity`;
- revised active symbolic field;
- trajectory metadata;
- resonance annotation;
- body/life/dream correlation;
- new Graphiti episode relations;
- reviewed insight candidate;
- teaching threshold flag;
- revised personal-field rendering state.

It does **not** mean:

- the user has a new fixed identity;
- the natal chart changed;
- the Jungian type changed;
- a Tarot reading became an identity claim;
- a dream became diagnostic fact;
- Nara overwrote M4-0.

## 4.3 Recommended Naming Clarification

To avoid future confusion, the system should distinguish:

```text
M4-0 Identity-System Branch
  the structured sources and computations of identity-system evidence

Q_identity contribution
  the stable identity-quintessence vector derived from reviewed identity-system outputs

#4.4.4.4 Personal Pratibimba
  the protected living personal field that composes Q_identity, Q_transit, and Q_activity

Q_composed
  the live runtime state of the personal field
```

So even if implementation language keeps `Q_identity(#4.0)`, the UX/system language should not suggest that M4-0 is “the user.” Better phrasing:

> **M4-0 contributes to Q_identity. #4.4.4.4 holds the living user-facing personal Pratibimba.**

# 7. M4.5 and the Epii Relay

## 7.1 M4.5 Is Not Only Review

M4.5 is review, but it is also the threshold where Nara may open Epii as canon teacher / paidagōgos.

M4.5 asks:

```text
Review:    Can this become stable meaning?
Pedagogy:  Does the user need or want the map here?
Relay:     Which coordinate/source/canon path should Epii reveal?
Return:    How does the teaching return to the user's life?
```

## 7.2 Epii's Role

Epii should not appear as an abstract lecturer.

Epii should function as:

> **the conductor toward the bimba map, corpus, canon, and teaching path.**

Inside Nara, Epii enters when:

- the user asks to understand the system;
- the user asks why a reading was made;
- the same pattern recurs and becomes a learning threshold;
- a review proposal requires canonical support;
- an oracle/dream/journal event clearly touches a major coordinate;
- the user wants a study path, practice path, or map traversal.

Epii should receive safe handles by default, not raw private content.

## 7.3 The Epii Relay Flow

```text
personal event
→ Nara reflection
→ M4-3 PatternPacket
→ M4.5 detects teaching threshold
→ Epii locates bimba/corpus/canon path
→ Epii teaches the map in relation to the user's need
→ Nara returns the teaching to body, life, dream, or practice
→ #4.4.4.4 receives reviewed/decayed update if appropriate
```

## 7.4 Teaching Thresholds

M4.5 should distinguish levels of pedagogical readiness:

```yaml
teaching_threshold:
  none: no teaching, only reflection
  explain: show why Nara read the event this way
  map: reveal the QL/MEF/bimba coordinate
  canon: surface source/corpus/canonical teaching
  curriculum: propose longer study/practice path
```

This keeps Nara from over-teaching.

# 8. Branch Interactions

## 8.1 M4-0 ↔ M4-3

Identity-system evidence provides stable symbolic background. M4-3 may compare live patterns against this background but cannot mutate M4-0.

Example:

```text

M4-3 sees recurring Water-heavy dream material.
Correct conclusion: possible compensatory pattern in personal Pratibimba.
Incorrect conclusion: user identity has changed to Water type.
```

## 8.2 M4-1 ↔ M4-3

M4-1 tests pattern in the body.

Example:

```text
Symbolic reading says “Fire is blocked.”
M4-1 asks: is there fatigue, inflammation, anger suppression, heat, coldness, collapse, or overdrive?
M4-3 integrates only what body/life evidence supports.
```

## 8.3 M4-2 ↔ M4-3

M4-2 provides oracle events. M4-3 integrates them into trajectory.

Example:

```text
Tarot gives Tower at P4.
I Ching gives moving line 4.
M4-3 asks whether this is a shared contextual-disruption pattern, or just a one-off oracle echo.
```

## 8.4 M4-3 ↔ M4-4

M4-4 provides lived evidence. M4-3 provides pattern logic.

Example:

```text
M4-4 shows three journal entries using images of freezing, waiting, and blocked speech.
M4-3 routes this through L3/L3' and L4/L4': process becoming lived evidence.
```

## 8.5 M4-3 ↔ M4.5

M4-3 detects when a pattern becomes review-worthy or teachable. M4.5 decides whether to open review or Epii relay.

Example:

```text
Repeated L1/L2 contradiction across journal, dream, and oracle.
M4-3 flags teaching_threshold: map.
M4.5 opens Epii to teach the 1/2 adjacency square.
```

## 8.6 M4.5 ↔ #4.4.4.4

M4.5 may approve an insight, playbook, curriculum, or personal-field update. The update lands in #4.4.4.4, not M4-0.

Example:

```text
Reviewed insight: “Conflict between surrender and control is a recurring practice threshold.”
Stored as protected personal-Pratibimba insight.
M4-0 identity-system files remain unchanged.
```

# 10. UX Rules and Guardrails

## 10.1 Never Mutate M4-0 from Daily Activity

Journal, dream, body, and oracle data may update:

- Q_activity;
- trajectory;
- active pattern field;
- Graphiti relations;
- personal-Pratibimba rendering;
- reviewed insight proposals.

They do not mutate M4-0 identity-system sources.

## 10.2 M4-3 Must Be Structured Before Conversational

M4-3 produces structured PatternPackets first.

Conversational Nara voice then interprets for the user.

This preserves the difference between:

```text
parser / pattern engine / evidence path
```

and:

```text
dialogic response / contemplation / practice suggestion
```

## 10.3 Symbolic Systems Are Dialects, Not Authorities

Tarot, I Ching, astrology, dreams, and Mahāmāyā do not individually determine meaning.

They become meaningful through:

- QL/MEF lens-position;
- M4-3 cross-dialect integration;
- M4-1 body accountability;
- M4-4 lived evidence;
- M4.5 review when needed.

## 10.4 The Body Has Veto Power

If the user is dysregulated, exhausted, grieving, panicked, or somatically overwhelmed, Nara should not open abstract teaching by default.

M4-1 can route the response toward:

- grounding;
- breath;
- rest;
- gentle reflection;
