# ADR-05-009 — Recast notice for ADR-001/002/003 (superseded by canon §2-§3)

**Status:** Recorded 2026-06-01
**Decision register:** PRD-01, PRD-02, PRD-03 (all resolved)
**Predecessors:**
- [ADR-001 — Theia runtime mode](../../../../Body/M/epi-tauri/decisions/adr-001-theia-runtime-mode.md)
- [ADR-002 — Single vs multi-webview](../../../../Body/M/epi-tauri/decisions/adr-002-single-vs-multi-webview.md)
- [ADR-003 — Bridge ownership](../../../../Body/M/epi-tauri/decisions/adr-003-bridge-ownership.md)

## Context

The three Track-05-T0 ADRs at `Body/M/epi-tauri/decisions/` were authored before the Theia-only canon revision ([m5-prime-system-shape-and-tauri-ide-canon.md §2-§3](../../../Bimba/Seeds/M/M5'/m5-prime-system-shape-and-tauri-ide-canon.md)). They answered PRD-01/02/03 under the working assumption that Tauri would be the canonical desktop wrapper. Canon §2-§3 has since retired the Tauri wrapper entirely. The pre-recast ADRs remain on disk as historical record; this ADR records the recast bridge so that future readers see the precise correspondence.

## Recast mapping

### ADR-001 — Theia runtime mode

| Pre-recast | Post-recast (canon §2-§3) |
|---|---|
| **Question:** Browser-mode-in-Tauri-webview vs Tauri-supervised localhost Theia vs Tauri-spawned Theia Electron sidecar | **Question:** Which Theia delivery target is canonical? |
| **Selected (Track 05 T2 amendment):** Option 1 — bundled browser-mode loaded via Tauri `pratibimba://` URI protocol | **Selected (canon):** Electron-app (canonical desktop) + browser-app (CI/Docker/headless), both from one `theia-app/` tree. No Tauri wrapper. |
| **Artifact:** [Tauri pratibimba prototype at commit `dd985f7`](../../../../Body/M/epi-tauri/decisions/adr-001-theia-runtime-mode.md#05t2-prototype-amendment-2026-06-01) | **Artifact:** [ADR-05-004](./adr-05-004-electron-target.md) + [ADR-05-005](./adr-05-005-electron-builder.md) + [ADR-05-006](./adr-05-006-docker-browser-mode.md) |

The dd985f7 prototype work is **architecturally obsolete** under canon §2-§3 but technically valid as a proof of concept. The `pratibimba://` URI handler, the `pratibimba_summon_ide` Tauri command, and the CSP additions in `tauri.conf.json` are migration-source-only material; none of them carries forward into `Idea/Pratibimba/System/`.

### ADR-002 — Single vs multi-webview

| Pre-recast | Post-recast (canon §2-§3) |
|---|---|
| **Question:** Inside the Tauri wrapper, does `/body` and `/pratibimba/system` share one webview (navigation) or two (multi-window + tray)? | **Question obviated.** No webviews. One Theia process. Two workspace layout modes (`daily-0-1.layout`, `ide-deep.layout`) switched via Theia's `Layout Restorer` / `Workspace` services. |
| **Selected:** Multi-webview + tray/background for `/body` | **Selected (canon):** Single Theia process; layouts switch intra-process. |
| **Artifact:** [ADR-002](../../../../Body/M/epi-tauri/decisions/adr-002-single-vs-multi-webview.md) | **Artifact:** Track 05 T2 deliverables (the two layout descriptors + omni-panel intent routing) — landing in T2. |

The layout descriptors `daily-0-1.layout` and `ide-deep.layout` are T2 deliverables under the recast plan; they replace what would have been two separate webviews.

### ADR-003 — Bridge ownership

| Pre-recast | Post-recast (canon §2-§3, PRD-03 resolution) |
|---|---|
| **Question:** Does `KernelBridgeAPI` live in Theia process, Tauri Rust process, or hybrid singleton? | **Question resolved:** Kernel-bridge is a first-loaded Theia extension. Its backend module connects to the external `Body/S/S3/gateway` Rust process via WebSocket/JSON-RPC; its frontend module publishes `KernelBridgeAPI` through Theia DI. |
| **Selected (provisional):** Hybrid — Tauri singleton + Theia-native KernelBridgeAPI adapter | **Selected (canon):** Theia-only ownership; gateway is external Rust process; no Tauri component. |
| **Artifact:** [ADR-003](../../../../Body/M/epi-tauri/decisions/adr-003-bridge-ownership.md) | **Artifact:** Track 05 T3 deliverables (the `kernel-bridge` Theia extension); existing scaffold at `Idea/Pratibimba/System/extensions/kernel-bridge-readiness/`. |

## Decision

**The pre-recast ADRs 001/002/003 stay in `Body/M/epi-tauri/decisions/` as historical record.** They are not edited or deleted. This ADR is the only durable cross-reference future readers need.

For any future architectural decision referencing PRD-01/02/03, the canonical answer is the canon-revision in [m5-prime-system-shape-and-tauri-ide-canon.md §2-§3](../../../Bimba/Seeds/M/M5'/m5-prime-system-shape-and-tauri-ide-canon.md) and [11-open-architectural-decisions.md PRD-01/02/03](../../../../docs/plans/2026-05-31-mprime-and-sprime-implementation-tracks/11-open-architectural-decisions.md). The pre-recast ADRs serve only as a historical record of how the question was framed before the canon resolved it.

## Consequences

- No code in `Idea/Pratibimba/System/` depends on the pre-recast ADRs.
- Cross-references in this workspace (`Idea/Pratibimba/System/docs/`) point at the post-recast ADRs (05-004 through 05-008).
- Any future audit of decision lineage can follow this ADR back to the original three.
