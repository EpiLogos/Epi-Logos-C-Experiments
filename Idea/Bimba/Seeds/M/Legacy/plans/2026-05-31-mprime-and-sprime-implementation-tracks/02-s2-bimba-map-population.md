# S2/S2' Bimba-Map Population Implementation Track

This track makes S2/S2' the early graph substrate for the M'/S' build: Neo4j holds the canonical Bimba coordinate graph, S2' supplies coordinate-aware graph law, and M5-3/M5-4 consume graph-backed coordinate, relationship, ontology, GDS, and retrieval data rather than local UI-only state.

## Goal

Deliver a production-ready S2/S2' Bimba-map population path for M5-2: a live Neo4j graph of canonical coordinate nodes and relationships populated from existing seeds/datasets, governed by the S2 schema registry, optionally lifted through n10s/OWL where canon and plugin availability support it, prepared for privacy-safe GDS overlays, and exposed through coordinate-native APIs for M5-3 rendering surfaces and M5-4 agentic mediation.

This plan does not implement `/Body` changes. It orders the engineering work that should follow.

## Architectural Keystones

- S2 is the graph of record for canonical Bimba coordinate topology. M5-3 surfaces and M5-4 agents may render, query, traverse, and mediate the graph, but they must not create renderer-local coordinate trees, relation laws, GDS recommendations, or ontology inferences.
- Coordinate identity is `:Bimba {coordinate}`. Legacy `#` and `bimbaCoordinate` data must resolve intentionally into the M-family branch without forking a second namespace. Compatibility labels/properties are migration aids, not new graph authority.
- The S2 registry must lead population. Every imported node property and relationship property must be registered in `graph-schema`; every production relationship type must be registered or explicitly marked compatibility-only before population becomes a release gate.
- Population order is schema first, seed second, dataset import third, semantic enrichment fourth, API consumption fifth. M0'/M1'/M2'/M3'/M5' should not depend on client-side placeholders while graph law is still unpopulated.
- Neosemantics/OWL is conditional but must be planned as first-class graph law. If n10s is available, S2 should maintain an `epi:` ontology for coordinate classes, relation families, M-family node semantics, source/spec/code/test anchors, and Anuttara syntax fields. If n10s is not available in the selected Neo4j topology, graph doctor must report OWL readiness as blocked rather than silently faking inferences.
- GDS readiness starts with Option 1 from [[S2-SPEC]] / [[S2-S2i-GRAPH]]: a canonical structural projection plus local active-coordinate personalization outside the canonical graph. Private Graphiti/Nara episode bodies are excluded from canonical GDS projections.
- The internal API surface is coordinate-native: `s2.graph.query`, `s2.graph.node`, `s2.graph.traverse`, `s2.graph.pointer_web.*`, `s2'.coordinate.resolve`, `s2'.retrieve`, `s2'.rerank`, and `s2'.enrich`. `bimba-mcp` remains useful for external MCP-facing access, but Pi/Anima/Sophia/Aletheia should consume gateway/service contracts rather than treating MCP as their internal organ.
- Readiness is more than service connectivity. `epi graph doctor` or its JSON successor must distinguish Neo4j up, schema present, seed present, dataset populated, registry clean, n10s/OWL ready, GDS ready, semantic embeddings current, Redis semantic-cache available, S1 queue drain available, and privacy-safe Graphiti boundary enforced.

## Tranches

1. **T0 - Authority and drift preflight.**
   Deliverables: inventory the live S2 graph modules and tests; record current schema version, seed count expectations, dataset plan, registry relationship set, Docker plugin state, and M5-3/M5-4 API consumers; define the exact live-test environment variables for Neo4j and Redis.
   Verification: `cargo test --manifest-path Body/S/S2/graph-schema/Cargo.toml`; `cargo test --manifest-path Body/S/S2/graph-services/Cargo.toml`; `docker compose -f docker-compose.epi-s2.yml up -d neo4j redis`; `epi graph doctor` reports Neo4j/Redis status without requiring mocked services.

2. **T1 - Schema and relation-law convergence.**
   Deliverables: reconcile `graph-schema` relationship registry with the relationships actually emitted by `seed.rs` and `dataset_import.rs`; either register production seed/import relationships such as `GENERATES`, `ENTANGLES`, `INTERLEAVES`, `MANIFESTS`, `BEDROCK`, `FAMILY_CONTAINS`, `MOBIUS_RETURN`, `ANCHORED_TO`, and deep dataset relation types, or migrate seed/import output to registered canonical relation types with evidence properties; add/confirm canonical node property specs for Anuttara fields (`c_1_symbol`, `c_1_formulation_type`, `c_1_complete_formulation` or their approved names), namespace/provenance fields, ontology readiness fields, and GDS overlay provenance.
   Verification: registry tests reject ad hoc relationship/property drift; a live seeded graph returns zero relationship types outside the registry or approved compatibility list via `CALL db.relationshipTypes()` plus a registry comparison; `validate_node_properties` and `validate_relationship_properties` pass on representative low-detail and deep dataset records before import.

3. **T2 - Neo4j topology and plugin readiness.**
   Deliverables: decide and implement the local service topology for APOC, n10s, and GDS readiness; update the operational runbook and doctor checks to report plugin/procedure availability; add a clear fallback state for environments where n10s/GDS are intentionally unavailable.
   Verification: live Neo4j procedure checks prove APOC is available and either prove `n10s.*` / `gds.*` are available or produce explicit blocked readiness states; `epi graph doctor` JSON includes separate `neo4j`, `schema`, `n10s`, `owl2_rl`, `gds`, and `privacy_projection` readiness fields; no test passes by stubbing plugin procedure results.

4. **T3 - Canonical coordinate seed baseline.**
   Deliverables: run schema creation and `seed_coordinate_space` idempotently; verify the base coordinate universe (# root, psychoids, weaves, CF nodes, family meta-nodes, 72 family coordinates, VAK nodes, pointer relations) is queryable by canonical coordinate APIs; resolve the root `#` to M-family migration behavior for API lookups and dataset imports; produce a seed snapshot query set for M0'/M1'/M5' smoke consumers.
   Verification: on a clean Neo4j volume, `epi graph init` or `epi graph bootstrap` creates constraints, indexes, vector index, expected seed node counts, and expected relationship counts; `s2'.coordinate.resolve` maps `#4` to `M4` while preserving intentional access to the psychoid/root facts where required; `s2.graph.node` returns real relations for sample coordinates `M0`, `M2`, `M4`, `M5`, `CF_FRACTAL`, and `CPF`.

5. **T4 - Dataset population from existing corpus.**
   Deliverables: import `docs/datasets/low-detail/` and all six deep branches (`anuttara-deep`, `paramasiva-deep`, `parashakti-deep`, `mahamaya-deep`, `nara-deep`, `epii-deep`) through `DatasetImporter`; preserve branch provenance (`c_3_source_dataset`, `c_3_dataset_branch`, `c_3_dataset_branch_label`); apply low-detail labels without creating ungoverned authority; produce a deterministic import report that names imported nodes, imported relations, skipped nodes, skipped relations, and skip reasons.
   Verification: a clean graph can run `epi graph import all` against `docs/datasets` and then pass Cypher checks for branch counts, missing endpoints, duplicate coordinates, raw `bimbaCoordinate` properties, unregistered properties, unregistered relationship types, and null relation endpoints; every skip is attributable to a real source-data issue or explicit migration rule; tests use the real dataset files or fixture slices generated from them, not invented demo data.

6. **T5 - Anuttara and OWL/neosemantics bridge (gated by T2).**
   Deliverables: define the `epi:` ontology source artifact and import/export path; lift coordinate classes, relation families, source/spec/code/test anchors, and Anuttara syntax fields into n10s where available; implement OWL2 RL readiness and SHACL/reporting checks as graph facts; expose provenance so M0' can show `symbol`, `formulation_type`, and `complete_formulation` only when S2 supplied or inferred them.
   Verification: live Neo4j n10s commands import the ontology and return inferred labels/classes for known sample nodes; M0' inspector contract tests fetch Anuttara fields from Neo4j/S2 response payloads and fail if renderer-local placeholder values are used; doctor reports missing n10s/OWL as blocked, not green.

7. **T6 - GDS baseline projections and tangent overlays (gated by T2/T4).**
   Deliverables: implement Option 1 GDS projection over canonical public Bimba topology; define projection names, included labels, included relationship types, and excluded protected namespaces; run FastRP, personalized PageRank, node similarity, Louvain/community, and KNN where plugin support permits; return derived overlay payloads with provenance, algorithm version, projection version, and privacy boundary status.
   Verification: `gds.graph.list` shows the expected projection on a populated graph; algorithm calls run against the real local projection or controlled graph fixtures loaded into Neo4j; overlay API responses for sample coordinates include derived status and never create canonical relationships automatically; tests prove private Graphiti/Nara labels or body properties are absent from GDS inputs and outputs.

8. **T7 - Coordinate-native graph API parity.**
   Deliverables: make the Rust S2 graph-service methods and S0 `epi graph` adapter line up with the canonical API names; expose `s2.graph.query`, `s2.graph.node`, `s2.graph.traverse`, `s2.graph.pointer_web.compute`, `s2.graph.pointer_web.refresh`, `s2'.coordinate.resolve`, `s2'.retrieve`, `s2'.rerank`, and `s2'.enrich` through the chosen gateway boundary; include source/spec/code/test anchors, pointer-web descriptors, ontology readiness, GDS overlay handles, disclosure density, and namespace in response envelopes.
   Verification: live API tests use Neo4j data seeded and imported by T3/T4; coordinate resolution, node lookup, relation traversal, GraphRAG retrieval, hybrid rerank, and enrichment all return real graph data with provenance; raw Cypher remains parameterized and read-only by default; M5-4 retrieval tests show Anima/Pi/Aletheia/Sophia can request coordinate pools without calling `bimba-mcp` as an internal dependency.

9. **T8 - M5-3 and M5-4 consumption contracts (handoff tranche).**
   Deliverables: publish the graph payload shapes needed by M0' graph view, M1' relation walker, M2' correspondence provenance, M3' graph/wheel dual surface, M5' graph namespace viewer, and M5-4 agentic control room; define kernel-bridge graph client expectations and readiness states; provide sample response fixtures captured from real local S2 responses for frontend/agent tests.
   Verification: M5-3 contract tests render filtered subgraph data, pointer anchors, Anuttara fields, GDS readiness, and source/spec/code/test anchors from captured real S2 responses; tests fail if UI code computes graph relations, OWL inference, GDS recommendations, codon/tarot/planetary mappings, or `#` to M resolution locally; M5-4 agent tests retrieve graph-backed context pools with namespace boundaries intact.

10. **T9 - Reconciliation, runbook, and release gate.**
    Deliverables: create the operational runbook for clean bootstrap, non-destructive update, destructive rebuild, dataset reimport, ontology reload, GDS projection refresh, doctor/readiness, and drift audit; define CI/nightly live-test profile expectations; add graph metadata/revision tracking so engineers can tell which seed/import/ontology/GDS versions are live.
    Verification: a fresh engineer can run the documented local flow from empty Neo4j volume to populated graph and receive a green readiness report; `epi graph reconcile` or successor reports no schema, seed, dataset, relation-registry, ontology, GDS, semantic embedding, or kernel-relation drift; release is blocked if live Neo4j checks are skipped or replaced with mock-only tests.

- S3/S3' runtime: Redis/RedisVL runtime residency, SpaceTimeDB `world_clock`, DAY/NOW/session handles, and Graphiti episodic handles must stay temporally owned by S3/S3' even when S2 stores graph facts or safe handles.

- Existing datasets: `docs/datasets/low-detail/`, `docs/datasets/*-deep/`, `docs/datasets/deep-property-map.md`, and `docs/datasets/migrations/2026-05-17-bimba-scaffold-to-bimba.cypher` are the starting population corpus and migration references.

## Open Decisions

- Should production seed relationships expand the schema registry to include current seed relation types, or should `seed.rs` migrate to the smaller canonical relation set? Current tests intentionally reject ad hoc relationship types, while current seed code emits several unregistered graph-structural relationships.
- What is the exact canonical handling of the root `#` node after `#0..#5` map into `M0..M5`? Current code preserves a `#` seed root while import/API migration paths convert `#` to `M`; this must be made explicit before dataset population is treated as canonical.
- Which property names are canonical for Anuttara language fields: public aliases (`symbol`, `formulation_type`, `complete_formulation`) or coordinate-prefixed registry keys (`c_1_symbol`, `c_1_formulation_type`, `c_1_complete_formulation`)? M0' can display aliases, but S2 must own the storage/provenance contract.
- Will the local Neo4j topology include n10s and GDS in the same service used by engineers, or should ontology/GDS run in optional profiles? The plan assumes doctor can report `blocked` honestly when plugins are absent.
- Does S2 persist GDS outputs as derived nodes/properties, return them as ephemeral API overlays, or both? Canon says recommendations must not become canonical relations automatically.
- When does Option 3 GDS enrichment supersede Option 1, and what evidence is required? Build Option 1 first unless a later canon decision changes the privacy boundary.
- Which topology approach is production for non-orientable traversal: Approach A coordinate-duplication/preprocessing, or deferred B/C graph modeling? Build Approach A first unless contradicted by a companion architecture decision.
- Is `bimba-mcp` staying purely external/MCP-facing, or must it be refactored to consume the Rust S2 graph-service contract to avoid divergent coordinate parser/schema behavior? Current TS parser still documents `#` syntax as deprecated/invalid, while Rust resolves `#` intentionally.
- What is the destructive rebuild policy for local/dev graph population? Engineers need a safe, explicit path for wiping test-owned graph data without risking shared or user data.
- What exact Graphiti/pratibimba handles may appear in S2 responses before S5 review? The line between public safe handles and protected-local episode data must be encoded before M5-4 agent retrieval is enabled.

## Success Criteria

- A clean Neo4j/Redis local stack can be bootstrapped, schema-created, seeded, populated from existing datasets, checked by doctor, and queried through S2/S2' APIs without mock data or renderer-local fallbacks.
- The populated graph uses `:Bimba {coordinate}` as identity, has unique coordinate/UUID constraints, a 3072-dimensional vector index, registered properties, registered relationship types or explicit compatibility exceptions, and no accidental `bimbaCoordinate` property drift.
- Legacy `#` coordinates resolve into the M-family branch consistently across Rust services, TypeScript/MCP surfaces, dataset imports, graph API responses, and M0' search/selection boundaries.
- M0'/M1'/M2'/M3'/M5' can consume graph-backed node, relation, pointer, provenance, ontology readiness, and GDS overlay data from S2/S2' contracts; they do not compute graph law, ontology inference, or GDS recommendations locally.
- M5-4 agents can retrieve coordinate-scoped context pools, traverse relationship neighborhoods, and receive source/spec/code/test anchors with namespace boundaries preserved across `bimba`, `gnosis`, `etymology`, and governed protected handles.
- n10s/OWL and GDS are either live and verified against real Neo4j procedures/projections or explicitly blocked in readiness output. There is no silent green state when plugins are absent.
- Tests include live Neo4j/schema/population/query verification under an explicit live-test profile, plus contract fixtures captured from real S2 responses for frontend/agent tests. Mock-only tests are not accepted as proof of readiness.
- Privacy tests prove private Nara/Graphiti bodies do not enter canonical graph projections, GDS inputs, public M' payloads, or M5 review evidence unless governed review/promotion explicitly permits safe handles or summaries.
