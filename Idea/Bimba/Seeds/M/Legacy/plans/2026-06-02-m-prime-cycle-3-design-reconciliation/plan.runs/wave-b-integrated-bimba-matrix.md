## Anomalies

### Contradictions

- **CONTRADICTION-B-1 (= DCC-01.2):** UX §7 declares M0' a full CRUD engagement system with edits propagating to M2'/M3'. M0'-SPEC restricts to route/review; surface enforces `mutatesGraphCanon: false`. S2 has `RelationshipManager::create_*` + `PromotionPlan` but S3 gateway exposes only `query/node/traverse`. **Decision needed** (already routed to Wave A M0 tranche 1.2).
- **CONTRADICTION-B-2 (= DCC-01.3):** UX §3 row 1 names properties `c_1_symbol`, `c_1_complete_formulation`, `c_1_form`. SPEC text uses `symbol` / `formulation_type` / `complete_formulation`. Inspector keys on `properties.symbol` etc. Naming round-trip not unified. **Decision needed** (already routed to Wave A M0 tranche 1.3).
- **CONTRADICTION-B-3:** UX §3.1 names two distinct relation families (structural skeleton + correspondential web) as a load-bearing M0-2' framing. Schema `RELATIONSHIP_TYPE_SPECS` declares only the structural skeleton; the correspondence family is admitted through `sanitize_rel_type` raw-import without family discrimination. Surface flat-lists both. **Decision needed** on whether to (a) extend schema with `c_1_relation_family` enum + register correspondence relTypes, (b) keep them as dataset-derived with a separate registry, or (c) preserve flat-list and downgrade the UX two-family claim.

### CODE-PENDING

- **CPB-1:** No image-asset property in S2 schema (`c_1_asset` / `c_1_image` / asset_handle / sigil / seal / glyph absent across `Body/S/S2/graph-schema/src/lib.rs`). UX §6 requires asset handles for decan seals, angel sigils, planet glyphs, tarot cards. Wave A M0 tranche 1.6 already owns this orphan-fill.
- **CPB-2:** S3 gateway exposes no S2-graph write methods (`gateway-contract/src/lib.rs` lines 139–141 enumerate only `s2.graph.{query,node,traverse}`). If DCC-01.2 resolves toward CRUD, gateway write-method registration is required.
- **CPB-3:** Correspondence-family relations land in Neo4j as raw `c_2_relation_type` strings without a `c_1_relation_family` discriminator. The `c_5_correspondence` / `c_5_correspondence_basis` keys (dataset_import.rs lines 1179–1180) are partial coverage, but no enumeration partitions {structural, correspondential, kernel-core, inferred}.
- **CPB-4:** GDS Option-1 projection (`s2_public_bimba_option1_v1`) defined as plan, but `gds_procedure_count()` will return 0 + `blocked_overlay_payload` until live GDS install + projection runner invocation. M0-3' synchronic community view is therefore in `gds_ready: false` state by default.
