## §3 — M5-1: Philosophical articulation of the Anuttara language

### §3.1 What lives here

The Anuttara language is **developed as a theory**, not just constructed as a technical artefact. Before any axiom is formally encoded, it begins as a philosophical articulation — a written-out exploration of what the proposed addition means, what it relates to in existing strata, what symbolic-numerical correspondences it carries, what archetypal-grammatical position it occupies.

M5-1 Canon Studio is the surface for this work. It carries:

- The **current canonical articulation** of the Anuttara language — written in philosophical-English alongside the formal notation
- **Active developer-conversations** that propose articulations (the user's exploratory writing, Sophia's responses, Anima's aesthetic-checks)
- **Crystallised theory** that has reached a stable articulation and is ready for M5-2 technical-construction
- **Deprecated articulations** with reasons-for-deprecation (the canon retains memory of its own evolution)

### §3.2 The developer-conversational mechanism

The user must be able to have ongoing developer-level conversations with the system that elaborate the Anuttara language. This is NOT casual chat — it's the deliberate work of articulating the system's deepest grammar through dialogue.

The mechanism: at M5-1, the user opens a **language-development conversation**. The conversation is held in a protected scope (this is canon-development work, not casual journal). Within the conversation:

- The user proposes an articulation — typically in philosophical-English with formal-notation embedded
- Sophia engages dialogically, drawing connections to existing M0 strata, suggesting refinements, surfacing what doesn't quite cohere
- Anima provides aesthetic-coherence checks — does the proposed articulation *feel right* against the language's existing resonances?
- The Logos Atelier (M5-5) may be invoked when etymological-grammatical archaeology is needed to verify a claimed correspondence
- The user can iterate, refine, abandon, or accept the articulation

When an articulation reaches stable form, it gets **deposited** as a candidate canon-addition. The candidate sits in M5-1's review-queue, visible to Sophia/Anima/Epii governance.

### §3.3 The crystallisation step

A candidate articulation that passes governance review crystallises into:

1. A **canonical-philosophical-text** at M5-1 (the articulated theory as written prose)
2. A **formal-notation entry** that gets queued for M5-2 technical-construction (the OWL/SHACL lifting work)
3. A **bimba-map node update** — the relevant M0 node's `symbol` and/or `formulation_type` property gets the new articulation written into it (via governed Cypher write through the S2 graph-services layer)
4. An **M5-0 library entry** — the articulation enters Epii's documentary holding for future RAG-retrieval

The crystallisation is the moment the developer-conversation work becomes part of the system's canon. Until that moment, the articulation lives as conversation-record; after that moment, it lives as canon.

### §3.4 Sensitive coordinate-class constraint

Some Anuttara coordinates are **structurally load-bearing** (the R-virtues at M0-2-9, the Siva logical operators at M0-5-(0/1), the M0-4 holographic-matrix base coordinates). Articulations that propose to modify these require **higher governance threshold** — typically explicit user-final-validation per the CLAUDE.md ur-process (Human = Ontological Vision + Final Validation).

The system surfaces this constraint at the articulation step: when a proposal touches a load-bearing coordinate, the M5-1 surface explicitly marks the proposal as "load-bearing modification — final validation required."

- **Specialized M0-internal classes**: `epi:VoidGrammarRoot` (for M0-0), `epi:BrimmingVoid` (for M0-1), `epi:Paramesvara` (for M0-2-9), etc.

#### §4.1.B Property characteristics

Relation types from the bimba map's `graph-schema/lib.rs` get OWL characteristics:

- `epi:invertsTo` — **symmetric, irreflexive** (X ↔ X' for any family-pair)
- `epi:bedrock` — **functional** (each family coord has exactly one psychoid bedrock per position)
- `epi:manifests` — **inverse of bedrock**
- `epi:contains` — **transitive** (parent-child coordinate nesting via `coordinate_parent`)
- `epi:partOf` — inverse of contains, transitive
- `epi:derivesFrom` — **transitive** (provenance chain)
- `epi:reflectsAs` — **symmetric** (Bimba ↔ Pratibimba via #5 → #0 Möbius)

- Target node-class (what does this virtue apply to)
- Property constraints (what conditions does the virtue require)
- Severity (info / warn / violation)
- A reference back to the canonical-philosophical-articulation at M5-1

Additionally, structural-invariant shapes:
- `epi:BedrockSinglenessShape` — every family coord has exactly one bedrock
- `epi:InversionPairCompletenessShape` — every coord has its inverse counterpart
- `epi:M04BaseExistenceShape` — every M-family subsystem has its M0-4 base coordinate
- `epi:M0CoreRelationsCountShape` — the 65 M0_CORE_RELATIONS skeleton is intact

#### §4.1.D n10s integration

The neosemantics (n10s) Neo4j plugin handles the RDF/OWL import-export pipeline:

```
[OWL .ttl files at canonical-source] 
   → n10s.graphconfig.init
   → n10s.rdf.import.fetch (import axioms into Neo4j)
   → n10s.rdf.export.cypher (export bimba-graph as RDF for reasoner)
   → [HermiT / Pellet / ELK reasoner runs]
```

The pipeline is maintained at M5-2 with versioning, rollback discipline, and audit trails.

This discipline is structurally why Anuttara sits at boundary-implicate position 0: its work is to **hold the frame**, which formal-construction does reliably and statistical-learning does not.

### §6.2 The agentic review workflow

When a proposal moves from M5-1 articulation to M5-2 construction, it passes through an agentic review workflow:

1. **Sophia review** — does the proposal cohere with existing canon? Are there tensions with prior articulations? Is the articulation philosophically rigorous?
2. **Anima review** — does the proposal feel right? Are there aesthetic resonance-violations?
3. **Pi translation-check** — can the proposal be translated cleanly to formal notation? Are there ambiguities?
4. **Atelier archaeology** (if invoked) — does the claimed etymological/grammatical correspondence hold under provenance-tracing?
5. **User final-validation** — for load-bearing coordinates, explicit user approval required

The workflow doesn't auto-promote. Every step produces an annotation; the user has visibility into all annotations; the user decides whether to proceed.

### §6.3 The dispatch-to-other-subsystems

These are coordinated through Pi's task-dispatch capability with provenance tracked back to the originating Anuttara articulation.

## §10 — Construction discipline summary

The work this spec covers is **construction**, not training. Concrete implications:

- Every change is a deliberate act under explicit governance
- The user's developer-level conversation is the primary input-channel
- Sophia/Anima/Pi/Aletheia agents support but do not auto-mutate canon
- All changes are reversible via deprecation-with-reasons
- Load-bearing coordinates (R-virtues at M0-2-9, Siva operators at M0-5-(0/1), M0-4 holographic-matrix bases) require explicit user final-validation
- The audit-trail is complete: every articulation traces from initial-conversation → review-annotations → crystallisation → bimba-map write
- The runtime-consultation surface (sensitive-context tracing API per §4.3) is **read-only** — it consults the constructed language; it doesn't mutate it
- Improvements that emerge from runtime observations (e.g., a pattern across many user oracle-readings suggesting a new R-virtue articulation) flow back as **conversation-proposals**, not as direct mutations

This is the structurally-correct discipline for the boundary-implicate position 0's relationship to its own development at the M5 operational-capacity surface.
