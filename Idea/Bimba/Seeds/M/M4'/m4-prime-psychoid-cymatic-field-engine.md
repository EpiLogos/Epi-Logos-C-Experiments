## §12 — The simulation register: full physics + caching/memoisation

The user's instruction is to go for **full simulation** while spec'ing for both registers (full simulation + cymatic-inspired-stylised). This section specifies the simulation register options and the optimisation strategy.

### §12.1 Two registers spec'd in parallel

**Option F — Full simulation**:
- Real wave-equation compute on a 3D voxel field
- Frequencies as actual sound waves producing actual standing-wave patterns via numerical PDE solver
- Highest fidelity (cymatic-physics-true)
- Highest compute cost (GPU-intensive; may not hit 60fps on lower-end hardware)
- Hardest to art-direct (parameters tuned by physics, not aesthetics)

**Option S — Stylised cymatic-inspired**:
- Standing-wave patterns generated by physics-informed shaders
- Aesthetic parameters depart from pure physics for readability
- Mid fidelity
- Mid compute cost (interactive on most hardware)
- Art-directable

**Architecture decision**: build both, with **runtime switching** based on:
- Device capability (probe at startup; auto-select)
- Engagement mode (contemplative-deep = Option F; ambient/journal-writing = Option S)
- User preference (explicit setting)

The engine layer is the same; only the field-compute shader-set differs between modes.

### §12.2 Caching / memoisation strategies

Full simulation is expensive. Optimisations:

**Stable-state caching**: when q_Nara and the active chakra-frequency set are unchanged over multiple ticks (the common case — chakra-planet mapping changes only on decan boundaries, every few hours), the cymatic field-pattern is stable. Cache the field once; only update on state-change events.

**Delta-update on small perturbations**: small changes in Q_composed (the common case as Q_activity perturbs the otherwise-stable state) produce small field-deltas. Compute deltas relative to cached stable-state rather than recomputing from scratch.

**Memoisation by quaternion-canonical key**: identical Q_composed values (within fixed-precision quantization per §11.4 of the integration paper) produce identical fields. Memoise field-renders by quaternion-canonical key. For a given user, most Q_composed values across a day cluster in a small region of S³ — high memoisation hit rate.

**Pre-computed transit window**: for the next 24 hours, cosmic-clock state is deterministic from Kerykeion. Pre-compute the cymatic-field background for each significant transit-window (decan changes, major aspect formations) during idle time; cache as recorded animations. Playback uses cached background + live Q_activity perturbation overlay.

**Audio-driven recomputation only**: the cymatic field is driven by the audio frequency-mix. If the audio mix hasn't changed (chakra-planet mapping stable), the field doesn't need recomputation. The audio is the *primary state*; the visual is *derived* from it; visual updates only when audio updates.

**Recorded contemplative-engagement-animations**: for common contemplative-engagement modes (deep contemplation, oracle-cast contemplation, dream-reflection), pre-record reference field-evolutions; the live render overlays user-specific modulation onto the reference animation. Hybrid approach: recorded-base + live-overlay.

### §12.3 Update-window strategy

Not every aspect updates at every frame:

| Aspect | Update frequency |

### §18.8 Möbius/Klein topology handling in GDS

Standard GDS random-walk algorithms (Node2Vec, FastRP, PageRank) don't natively handle the `INVERTS_TO` semantics — the Möbius-twist where traversing X → X' should flip an inversion-phase bit on the walk. Three handling approaches:

**Approach A — Coord-duplication preprocessing**:
- Duplicate every coord into (phase=0, phase=1) versions
- `INVERTS_TO` becomes a regular edge between phase=0 and phase=1 copies
- Run standard GDS on the doubled graph
- Collapse phase=0 and phase=1 embeddings post-process via averaging or PCA
- **Pro**: Standard algorithms work unmodified
- **Con**: Doubles graph size; loses semantic meaning of "phase-flip on traverse"

**Approach B — Custom Klein-aware walks**:
- Implement custom walk that tracks phase-state and flips on `INVERTS_TO` traversal
- Embeddings naturally encode the Klein topology
- **Pro**: Topologically honest
- **Con**: Implementation work; non-standard

**Approach C — Two embeddings per node**:
- Run standard GDS but produce two embeddings per coord: one for phase=0 walks, one for phase=1 walks
- Recommendation queries select embedding by current user-phase-state
- **Pro**: Simpler than custom walks
- **Pro**: Phase-aware queries are direct
- **Con**: Per-coord storage doubles; phase-selection logic needed at query-time

**Recommendation in spec**: Approach A for first prototype (proves the principle works); Approach B or C for production if Klein-topology effects on recommendations are pronounced enough to matter. The Klein topology is load-bearing per `M1'-SPEC` §10 (K² double-cover of chromatic fifths); honest GDS handling matters at scale but can be deferred for early prototypes.
