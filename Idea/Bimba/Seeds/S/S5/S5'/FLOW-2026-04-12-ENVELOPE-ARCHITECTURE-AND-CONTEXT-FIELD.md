Its purpose is to define the next architectural seam:

1. the full layered shape of the envelope that will govern [[Day]] / [[NOW]] task runs
2. the relation of that envelope to the coordinate families and `/Idea` residency
3. the way working inspirations such as `claude-memory-compiler`, `epi-claw` / `OpenClaw`, and `claw-code-parity` are to be structurally assimilated rather than copied

- the envelope is not just a packet of metadata
- it is the formal articulation of how runtime transport, temporal legitimacy, coordinate law, filesystem residency, contextual economy, lived execution, episodic return, and system self-improvement remain one process

## #4 CONTEXT

The envelope must be layered because the architecture itself is layered.

If all concerns are collapsed into one object, the system loses precision:

- gateway transport starts pretending to be ontology
- runtime state starts pretending to be canon
- retrieval starts pretending to be context
- reflection starts pretending to be planning
- improvement starts pretending to be ordinary execution

The envelope therefore needs distinct seams.

These seams are not arbitrary.
They correspond to the differentiated life of the system:

- `[[S0]]/[[S0']]` — low-cost CLI and reflective ground access
- `[[S1]]/[[S1']]` — lawful filesystem materialisation and content membrane
- `[[S2]]/[[S2']]` — graph, Redis, and structured retrieval body
- `[[S3]]/[[S3']]` — transport, gateway, shared runtime, and temporal surfacing
- `[[S4]]/[[S4']]` — ta-onta execution and agentic inhabitation
- `[[S5]]/[[S5']]` — return, recollection, crystallisation, and improvement

This layered shape should also make future tabulation easier.
Each seam below is written with its likely coordinate homes and subsystem affinities made explicit where possible.

### 1. TRANSPORT ENVELOPE

This is the leanest layer.
Its purpose is carriage, not truth.

This is where the `epi-claw` and wider `OpenClaw` lineage is genuinely useful.
The working patterns there are:

- stable `session_key` versus runtime `session_id`
- channel and requester origin separation
- session patch / reset / history semantics
- agent-scoped routing
- transport-facing history windowing

This layer should therefore carry only what the transport plane truly needs:

- `session_key`
- `session_id`
- `request_id`
- requester or origin handle
- channel or delivery surface
- thread or conversation scope
- target agent, team, or swarm
- history limit
- patch / reset lineage

The main law here is:

- `[[S3]]` and `[[S3']]` may carry the run
- they must not become the source of canonical meaning

So this layer is transport truth, not ontological truth.

### 2. RUNTIME ENVELOPE

This is the real execution substrate.
It is closer to `[[Nous]]` and `[[Techne]]` than to gateway transport.

This is where the `claw-code-parity` patterns are stronger than the `epi-claw` ones:

- explicit bootstrap context
- clear runtime and session-store separation
- permission and tool boundary handling
- transcript or state persistence
- runtime observability modes

This layer should hold:

- runtime bootstrap context
- workspace and source-root context
- session persistence handle
- permission and capability boundary
- observability versus headless mode
- terminal substrate such as `tmux` / `cmux`
- planned web or app hooks where real

This is where `[[Techne]]` belongs in the replanned architecture:

- not as a sovereign worker
- not as fake web capability
- but as a bounded helper that helps [[Nous]] and the preparatory arc set up a real environment

### 3. TEMPORAL ENVELOPE

This is Janus' seam.

The run is not abstract.
It is born within:

- a `[[Day]]`
- a `[[NOW]]`
- an episode boundary
- a temporal horizon

This layer should unify:

- `day_id`
- `now_id`
- `now_path`
- session start boundary
- session close boundary
- archive and continuation status
- cron or scheduled lineage where present
- kairos and `tick12` state where present
- Graphiti arc legitimacy

This layer is where:

- [[Psyche]] receives temporal ground
- [[Janus]] legitimates transition and threshold
- [[Chronos]] can act as a system feature rather than mere scheduler

This layer should stay close to `[[S3']]`, `[[S4']]`, and `[[S5.0']]`.

- what residue is expected?
- what manifestation family should receive the result?

This layer should therefore carry:

- primary family target such as [[S]], [[M]], [[P]], [[L]], [[C]], or [[T]]
- prime family target such as [[S']], [[M']], [[P']], [[L']], [[C']], or [[T']]
- any known subcoordinate targets
- governing `[[CPF]]`, `[[CT]]`, `[[CP]]`, `[[CF]]`, `[[CFP]]`, and `[[CS]]`
- guardrail families
- residue families
- manifestation families

This is how the architecture remains coordinate-aware rather than generic.

It also makes eventual population of `[[S Coordinate Lattice Scaffold]]` and later broader coordinate tabulations much easier.

- Agora-led external and common-square retrieval where required

This is the notebook-style economic pool:

- not yet the inhabited run
- not yet the canonical record
- but the provisional field from which the run may actually live

This layer should carry:

- source set
- scope set
- relevance grounds
- retrieval mode
- disclosure density
- project and task horizon

- episodic recall handles

This layer lives mainly across `[[S2]]`, `[[S2']]`, `[[S4']]`, and `[[S5']]`.

### 11. IMPROVEMENT ENVELOPE

This is the `[[S5]]` and autoresearch seam.

The architecture needs an explicit loop for bounded improvement, not a vague hope that good ideas will accumulate.

Two major loops now stand out.

#### DEVELOPMENTAL LOOP

This is closest to autoresearch:

- [[Sophia]] recognises a developmental vector
- [[Zeithoven]] shapes a challenger or next-form
- a bounded trial is run
- an evaluation surface judges baseline versus challenger
- the result is preserved, rejected, or folded back for another pass

This loop is especially useful for:

- `[[S]]/[[S']]` stack and substrate improvement
- `[[M]]/[[M']]` representational or subsystem evolution
- `[[T]]/[[T']]` better crystallisation methods

#### PRECISION LOOP

This is closer to Darshana:

- [[Darshana]] reads exactly
- [[Anansi]] differentiates source and placement exactly
- a bounded correction is proposed
- verification tests the correction
- only then is the correction allowed to manifest

This loop is especially useful for:

- `[[C]]/[[C']]` ontology and VAK precision
- `[[S']]` implementation-law alignment
- source-web and wikilink discipline
- graph relation precision

This layer should therefore carry:

- improvement target family
- baseline artifact or state
- challenger artifact or state
- evaluation surface
- keep or discard rule
- promotion destination
- residue class for later learning
