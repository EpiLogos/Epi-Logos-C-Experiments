# [[S3]]: Gateway Control Plane

> The sovereign personal interface of the stack: the plane through which participants act, sessions are governed, and runtime decisions are made.

## #5 Integration — What This Layer IS

[[S3]] is the imperative plane of the stack. It is the layer where command becomes lawfully actionable: the place where a participant, operator, or client addresses the system and expects a governed response. Its task is not merely transport. Its task is sovereignty at the level of invocation.

[[S3]] therefore owns the stable [[WebSocket]] and [[RPC]] command surface, with [[Epi-Claw Parity]] as a hard compatibility requirement rather than a vague aspiration. In concrete terms, the gateway is the runtime in `epi-cli/src/gate/` that accepts socket connections, emits a `hello-ok` frame, negotiates protocol/features, requires a `connect` handshake, validates method payloads, executes gateway-owned mutations, emits ordered events, and returns structured responses or structured errors.

### Gateway Surface

The gateway surface is not one command. It is a method manifest. The current parity canon groups that manifest into real families:

- connection and transport: `connect`, protocol version negotiation, challenge/nonce auth flow, reconnect, heartbeat, tick, gap-aware event delivery
- agent runtime: `agent`, `agent.wait`, `agent.identity.get`, `agents.list`
- chat runtime: `chat.send`, `chat.abort`, `chat.history`, `chat.inject`
- session authority: `sessions.list`, `sessions.preview`, `sessions.resolve`, `sessions.patch`, `sessions.reset`, `sessions.delete`, `sessions.compact`
- config and system control: `config.get`, `config.schema`, `config.set`, `config.patch`, `config.apply`, `status`, `health`, `status.summary`, `health.snapshot`, `presence.list`
- skills and model surfaces: `skills.status`, `skills.bins`, `skills.install`, `skills.update`, `models.list`, `usage.status`, `usage.cost`
- channels and temporal control: `channels.status`, `channels.logout`, `cron.list`, `cron.status`, `cron.add`, `cron.update`, `cron.remove`, `cron.run`, `cron.runs`, `wake`
- node, device, browser, and approval control: `node.*`, `device.*`, `browser.request`, `web.login.*`, `exec.approval.*`, `exec.approvals.*`
- maintenance and auxiliary operator lanes: `logs.tail`, `update.run`, `wizard.*`, `last-heartbeat`, `set-heartbeats`, `system-presence`, `system-event`, `talk.mode`, `tts.*`, `voicewake.*`

Its canonical event fanout is likewise concrete rather than symbolic: `agent`, `chat`, `tick`, `health`, and `heartbeat` are named event families with sequence semantics and reconnect implications.

### What `[[Epi-Claw Parity]]` Actually Means

`[[Epi-Claw Parity]]` does not mean "roughly similar to the old gateway." It means the Rust `[[S3]]` plane must preserve the behavioral contract expected by the existing `epi-claw` and [[OmniPanel]] clients:

- the same handshake shape: socket opens, `hello-ok` advertises methods/events, `connect` is first-class rather than optional
- the same session-facing affordances: clients can resolve, patch, delete, compact, and inspect sessions through the gateway rather than through hardcoded app assumptions
- the same operational domains: chat, agent runs, channels, skills, cron, config, nodes, devices, approvals, browser flows, logs, and maintenance surfaces are gateway-owned RPCs
- the same client compatibility pressure: the Electron controller contract and OmniPanel parity matrix are treated as authority references, not as disposable legacy code
- no fake completeness: if a surface is not real, it should be absent or explicitly unsupported rather than claimed under a parity banner

In other words, parity is behavioral equivalence of the command plane, not textual reuse and not nostalgic naming. The point is that an `epi-claw`-shaped client can talk to Rust `[[S3]]` and receive the same category of capability with truthful semantics.

[[S3]] is also the authority layer for [[Session]] identity. It is where a session is bound to canonical key, aliases, [[day]] identity, NOW path, workspace root, bootstrap scope, and subagent lineage. A session is not first real because it is displayed somewhere; it is first real because [[S3]] has accepted and governed it.

### Session Authority Means Real Identity Law

The gateway does not merely store chat transcripts. It governs session identity. The current parity model makes the compatible canonical session key primary on the wire and in storage, while allowing richer NOW-native identity to appear as alias and metadata. The session record expected by gateway-facing clients includes fields such as:

- canonical key and alias
- human label
- active agent identity
- subagent lineage
- workspace root
- bootstrap scope
- team and orchestration metadata
- NOW-facing lineage such as `sessionId`, `dayId`, and `vaultNowPath`

This is why `[[S3]]` is the authority plane for agent/session/runtime control. It decides which session a run belongs to, what subagent may inherit or spawn from what parent, what workspace and bootstrap scope apply, and what transcript/history belongs to that lineage. `[[S3]]` is where those rules are enforced.

The essential boundary is strict:

- imperative command belongs to [[S3]]
- shared synchronized world-state belongs to [[S3']]

This means [[S3]] validates, routes, persists, emits, and governs, but does not pretend to be the whole world. It is the control plane, not the inhabitable field. It owns command execution first: run registration, transcript writes, session mutation, terminal snapshot caching, event emission, and operator-facing truth about the runtime. `[[S3']]` only takes over once those imperative consequences must be rendered as shared state.

It also means that [[OmniPanel]] compatibility is not decorative. The gateway must stay readable by the existing client stack: renderer controller expectations, main-process gateway client behavior, and parity tests are part of the shape of `[[S3]]` itself. The gateway is PI-open and VAK-open because it must support active-agent switching, subagent participation, future signature/lane metadata, and richer invocation surfaces without collapsing the command contract.

## Cross-Layer Actualisation

[[Chronos]] actualises the temporal face of [[S3]] by giving command a lawful cadence, schedule, and recurrence surface. [[Khora]] actualises identity and bootstrap law by binding session authority to root, day, and NOW inheritance. [[Hen]] actualises content authority by ensuring that gateway-bound identity corresponds to real vault structures and writable worlds. [[Anima]] actualises orchestration by turning gateway authority into dispatchable multi-agent action. [[Aletheia]] actualises reflective return by carrying the consequences of command toward the shared-state plane of [[S3']].

## #0 Ground — What Returns to Source

The open questions of [[S3]] are structural, not definitional:

- when NOW-native naming should become the primary session identity rather than an attached aliasing layer
- where the public and remote exposure boundary of the gateway should finally sit
- what must remain inside [[epi gate]] and what should be lifted into richer clients such as [[OmniPanel]]
- how much maintenance and observability law belongs intrinsically to [[S3]] rather than to adjacent layers

Older [[PAI]], plugin, and harness readings remain part of the genealogy of the coordinate, but they are no longer the target shape. The target shape is settled: [[S3]] is the gateway control plane.
