# [[S4']]: Ta-Onta Extension Architecture

> The inverted (operational/API) layer of [[S4]]. Six-class PI extension architecture providing coordinate semantics to the agent runtime. Where [[S4]] is what the agent layer IS (technology), [[S4']] is what it DOES (orchestration, bootstrap, content coordination, temporal management, knowledge crystallisation).

## #5 Integration — What This Layer IS

[[S4']] is the **[[ta-onta]] extension package** — six named functions that together constitute the agent runtime's operational API. They are coordinated parts of ONE agent runtime, interacting through shared filesystem, shared services, and shared hook seams.

### The Principle: Named Functions, Not Separate Systems

Each extension folds its corresponding S-layer into the agent runtime:

| Extension | Position | S-Layer Fold | Responsibility |
|-----------|----------|-------------|---------------|
| **[[Khora]]** | S4-0' (#0) | [[S0]] Terminal/CLI | Bootstrap spine. Session identity. ALL vault writes. Secrets materialisation. |
| **[[Hen]]** | S4-1' (#1) | [[S1]] Obsidian/Vault | Content authority. [[CT]] templates. Frontmatter schema. Vault-to-graph sync. |
| **[[Pleroma]]** | S4-2' (#2) | [[S2]] Neo4j/Graph | Bounded primitives. Tool registration. Execution substrate. |
| **[[Chronos]]** | S4-3' (#3) | [[S3]] Gateway/Transport | Temporal authority. Day/NOW lifecycle. Cron. [[Kairos]]. |
| **[[Anima]]** | S4-4' (#4) | [[S4]] Agent (self-fold) | Orchestration centre. [[VAK]] evaluation. [[CF]] dispatch. Thread execution. |
| **[[Aletheia]]** | S4-5' (#5) | [[S5]] Sync/Integration | Crystallisation mode. [[Gnosis]] pipeline. Thought routing. SEED generation. |

### Internal Structure per Module

Each extension has exactly **3 folders** plus its core files:

```
{module}/
├── extension.ts       ← PI tool registrations
├── CONTRACT.md        ← Architectural contract
├── Sx/                ← Raw primitives (implementation)
├── Sx'/               ← QL augmentation (skills, agents)
└── M/                 ← Cross-cutting M-branch concerns
```

### S4-3': [[Chronos]] — Temporal Pathing and Day/NOW Lifecycle

**QL Position:** #3 ([[Mahamaya]] — pattern, process, temporal structure)

**Registered Tools:** `chronos_day_init`, `chronos_now_init`, `chronos_archive_day`, `chronos_cron_register`/`chronos_cron_list`, `chronos_kairos_fetch`/`chronos_kairos_status`, `chronos_temporal_status`, `chronos_graphiti_day_arc`, `chronos_decan_check`

**Key Concepts:**
- **SEED.md** = morning-context package crystallised by [[Aletheia]] evening cycle, consumed at 6 AM. Location: `/Idea/Empty/Present/SEED.md`. Contains yesterday's P5' insights reframed as today's P0' questions, carried-forward tasks from TOMORROW.md, and [[Aletheia]]'s Mobius cycle synthesis.
- **Archive path:** `{YYYY}/{MM}/W{WW}/{DD}/` — W{WW} mandatory
- **Archive guard:** `c_5_reflection_complete: true` required or `--force`
- **[[Chronos]] TRIGGERS creation events; [[Hen]] DEFINES structures; [[Khora]] WRITES**

**[[Kairos]] Integration — Three Temporal Modes:**

| Mode | Source | Description |
|------|--------|-------------|
| **Natal** | [[PASU]].md birth data → [[Kerykeion]] | Cosmic state at birth. `sun_degree_anchor`. Stable identity ground. |
| **Real-time** | Current planetary positions | Live transit layer. Daily kairos. (Future: live kerykeion polling) |
| **Kairotic** | Oracle consultation moment | Degree at oracle cast time. 4h decay. (Future) |

- **[[Kerykeion]]** (Python, backed by pyswisseph): computes natal chart + planetary degrees from `c_0_birth_date` + `c_0_birth_location` in [[PASU]].md
- **Degrees in SU(2) double cover:** 0-719 range (×2), `planet_degrees[10]` canonical mod-10 (Sun=0 through Pluto=9). Current implementation: 7 planets (Sun–Saturn, `planet_valid: 127`); 3 outer planets (Uranus/Neptune/Pluto) pending.
- **FR-3 compliance:** [[Kairos]] is additive, never a hard dependency. M4 works at 0 planets (`planet_valid = 0x00`). Feature-flagged via `KAIROS_ENABLED` env var.

**[[Graphiti]] Day Arc Lifecycle:**
- `chronos_day_init` → opens `"day:{day_id}"` arc (ql0, cpf:`(00/00)`, ct:2) with optional `kairos_snapshot`
- `chronos_archive_day` → closes day arc (ql5, cpf:`(5/0)`, ct:1) with optional `crystallisation` text
- Non-fatal if [[Graphiti]] sidecar not running

**Decan Boundary Checking** (cron every 2h via `chronos_decan_check`):
- Sun decan change (~10 days): close prior `sun-decan:{prev}` arc (ql5), open new arc (ql0) with decan_ruler, element, body_zone, hexagram_range
- Moon decan change (~2.5 days): ql3 episode with content `"moon enters {decan}: {ruler}, {body_zone}"`, cpf:`(0/1/2)`
- Checks against Redis decan state cache (`kairos:current_sun_decan`, `kairos:current_moon_decan`)

**Three Temporal Axes in [[Graphiti]]:**
1. **Calendar Time:** Day Sagas opened/closed by [[Chronos]]
2. **Astrological Time / [[Kairos]]:** Every episode carries sun_degree, moon_degree, planet_degrees[10], sun_decan, moon_decan
3. **Spanda Time:** tick12 (0-11) mapped to QL substages (#0 seed → #5 integration)

**Status:** Well-developed. Day-init includes SEED.md injection into `## #0 Question`. Archive with wikilink-preserving moves. Decan boundary check polls [[Graphiti]] sidecar. Kairos natal mode implemented in Rust CLI (`epi-cli/src/vault/kairos.rs`). Real-time and kairotic modes are future work.
