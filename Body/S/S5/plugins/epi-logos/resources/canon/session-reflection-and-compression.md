# Session Reflection and Compression

Near compaction, handoff, or session completion, the work needs to be reflected on and compressed before it is left. Raw sprawl is not a handoff. It is noise that the next cycle has to clear before it can begin.

## Why Compression Matters

Active runs generate material. Observations, traces, challenges, discoveries — the artifact field accumulates. If that accumulation is left as-is, the next cycle (whether in the same session or a new one) has to reconstruct what the run actually found rather than building from it.

Compression is not summarizing. It is a reflective act that asks: what genuinely cohered in this run? What remains paradoxical and must stay open? What failed, and what does that failure actually show? The answer to those questions is what gets carried forward — not the raw material that produced them.

## The Compression Sequence

1. **Review the active task or subtask.** What was actually being pursued? Has it been completed, partially completed, or abandoned?

2. **Review the active artifact families.** Which families moved? Which contain live material versus noise? What is actually in `Discovery/` that survived verification?

3. **Identify what became coherent.** What can now be stated clearly that could not be stated at the start? What holds and will continue to hold?

4. **Identify what remains paradoxical.** What is still genuinely unresolved? This is not failure — some questions need to stay open. But they should be named precisely, not just left implicit in the raw notes.

5. **Identify what failed and what it shows.** If part of the run failed, that failure is information. Compress it honestly. Name the next opening it reveals.

6. **Write or state the durable synthesis.** Objective work compresses toward `Thought/Insight/`. Subjective work compresses toward `Thought/Releasement/`. Mixed work often requires both.

7. **Leave the next cycle a clean entry point.** The compression should be written so that someone (or the same agent in a new session) can read it and know immediately: what is resolved, what remains open, and where the next run enters.

## Compression Targets

- Objective work → `Thought/Insight/`
- Subjective work → `Thought/Releasement/`
- Mixed work → both, with each carrying the appropriate face of the synthesis

## What Good Compression Looks Like

Good compression is specific. It names what cohered, not just "progress was made." It names what remains open, not just "there is more to explore." It tells the truth about failed runs rather than decorating failure as insight.

Good compression is honest about paradox. If a contradiction remains after the run, the compression should name it clearly rather than smoothing over it. Premature resolution in the compression stage is as bad as premature synthesis in the run itself.

Good compression is usable. The next cycle should be able to enter from it without having to reconstruct the context from scratch.

## When to Run the Compression Skill

The `compress-thought-artifacts` skill handles the executable side of this. Invoke it when:

- a task is ending
- a subagent is handing back work
- context compaction is imminent
- a cross-session handoff needs to be durable
