### D.2 Derivation Rules (The Algorithm)

For a given node N with parsed coordinate (family F, ql_position Q, inverted I):

#### Category 1: Family Refs (6)
Maps the node's QL position across all six families.

```
For each family X in {C, P, L, S, T, M}:
    X_ref = "{X}{Q}"     (if N is non-inverted)
    X_ref = "{X}{Q}'"    (if N is inverted)
```

Example: `M3` -> `c_ref = C3`, `p_ref = P3`, `l_ref = L3`, `s_ref = S3`, `t_ref = T3`, `m_ref = M3`

For sub-coordinates (e.g. `M0-2-4`), family refs point to the base position only: `c_ref = C0`, since sub-coordinates are M-branch specific and don't have cross-family equivalents at depth > 0.

#### Category 2: Reflective/VAK Refs (6)
Each reflective coordinate is a specific combination of families. The VAK-to-C mapping:

| VAK Coord | Composition | Maps to | Archetype |
|-----------|-------------|---------|-----------|
| CPF | Category-Position-Frame | C0 / S4-0' | Ground context |
| CT | Content-Type | C1 / S4-1' | Definitional freeze |
| CP | Context-Position | C2 / S4-2' | Positional frame |
| CF | Context-Frame | C3 / S4-3' | Lemniscate anchor |
| CFP | Context-Frame-Position | C4 / S4-4' | Fractal doubling |
| CS | Context-System | C5 / S4-5' | System synthesis |

```
For node with ql_position Q:
    cpf_ref = "S4-0'" (or the CPF node coordinate)
    ct_ref  = "S4-1'" (or the CT node coordinate)
    cp_ref  = "S4-2'" (or the CP node coordinate)
    cf_ref  = "S4-3'" (or the CF node coordinate)
    cfp_ref = "S4-4'" (or the CFP node coordinate)
    cs_ref  = "S4-5'" (or the CS node coordinate)
```

Reflective refs are positionally invariant: every node at any position points to the same 6 VAK coordinates. The specific VAK coordinate that "governs" the node's position is `VAK_NAMES[Q]` — this is the node's **primary reflective context**.

#### Category 3: Inversion Refs (6)
The polar opposite across each family via the `#` operation.

```
For each family X in {C, P, L, S, T, M}:
    X_inv_ref = "{X}{5-Q}'"   (if N is non-inverted: flip position, mark inverted)
    X_inv_ref = "{X}{5-Q}"    (if N is already inverted: flip position, remove inversion mark)
```

Example: `M2` -> `c_inv_ref = C3'`, `p_inv_ref = P3'`, etc. (position 5-2=3, prime added)
Example: `M2'` -> `c_inv_ref = C3`, `p_inv_ref = P3`, etc. (position 5-2=3, prime removed)

#### Category 4: Position Refs (6)
What the node IS when viewed from each QL position. These refs point to the node's own family but at each of the 6 base positions, creating a "positional projection" web.

```
For each position P in {0, 1, 2, 3, 4, 5}:
    p{P}_ref = "{F}{P}"    (node's own family, each QL position)
```

Example: `M3` -> `p0_ref = M0`, `p1_ref = M1`, `p2_ref = M2`, `p3_ref = M3`, `p4_ref = M4`, `p5_ref = M5`

For sub-coordinates, position refs stay at the parent family level — sub-coordinates inherit the positional web of their base coordinate.

## Phase E: Deep Dataset Import

### E.1 Scope

Deep import ingests the full M-branch dataset JSON files with ALL properties, not just the low-detail subset (name, description, essence, core_nature). Must follow the same updated cypher and coordinate rules established during low-detail import cleanup.

### E.2 LLM-Guided Property Assessment (Pre-Import Step)

**Before automating property mapping, we must be thoughtful about what each property SHOULD be at the coordinate level.** This is not blind automation of arbitrary props — each property must earn its place in the coordinate-driven schema.

For each dataset property encountered during deep import:

1. **Assess coordinate placement**: Use LLM to determine which `{family}_{position}_{semantic}` the property genuinely belongs to. A property's coordinate home reflects its ontological role, not just a convenient bucket.
2. **Filter vs. scaffold**: Some dataset properties are ephemeral/tooling-specific and should NOT become permanent graph properties. LLM evaluates whether each prop is genuinely coordinate-level data.
3. **Semantic validation**: Verify that the property's semantic content matches its coordinate placement. E.g., something about pattern/identity belongs at position 3, not position 1.
4. **q_* prefix assessment**: Dynamic `q_*` properties need review — they were generated by prior LLM passes and may contain redundant, misplaced, or poorly-named entries. Don't blindly passthrough.

This step produces a **validated property manifest** per dataset branch before any import runs.

## Phase F: Pointer Refresh

### F.1 Rationale

After deep import (Phase E), new nodes exist that were not present during Phase D. The pointer web must be recomputed so that:
1. New nodes get their 36-pointer web computed
2. Existing nodes' pointers that previously resolved to `None` (missing target) may now resolve to real coordinates
3. Relationship edges (POLAR_OPPOSITE, CONTEXT_FRAME_REF, HAS_LENS) are created for newly resolvable targets

### F.2 Implementation

Current executable method surface:

```
s2.graph.pointer_web.compute   # connection-free compute for agent/API orientation
s2.graph.pointer_web.refresh   # parameterized Neo4j write for one coordinate
```

`PointerWebRefreshPlan` writes the S2-owned properties:

- `c_5_pointer_web_json`
- `c_5_pointer_count`
- `c_5_pointer_family_refs`
- `c_5_pointer_reflective_refs`
- `c_5_pointer_inversion_refs`
- `c_5_pointer_position_refs`
- `c_5_pointer_lens_refs`

Future ergonomic CLI commands can batch over these same methods:


The `--diff` flag queries existing pointer values before computation, then reports:
- New pointers (was null, now resolved)
